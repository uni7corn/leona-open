# Leona Android SDK v0.4 Release Checklist

This checklist is the public-safe release review entrypoint for the Android SDK
v0.4 commercial pilot track. It complements the v0.4 release notes draft and
the evidence/privacy boundary document.

The SDK remains evidence-only. It collects and reports device/environment
evidence, returns BoxIds and diagnostics, and leaves final business decisions to
the customer backend.

## 1. Local Public-Safe Gate

Run the aggregate gate from the Android SDK directory:

```bash
./scripts/verify-v0.4-release-readiness.sh
```

Required result before release candidate review:

- status is `local-pass-with-external-blockers`
- failures are `none`
- secret values printed is `no`
- every blocker is explicitly external

The gate must not start paid devices, launch WeTest sessions, print provider
credentials, print Maven Central credentials, or embed customer secrets.

## 2. Public Archive Dry Run

Run:

```bash
./scripts/verify-v0.4-public-archive.sh
```

Required result:

- status is `pass`
- archive and manifest are written under `/tmp` or another explicit output
  directory
- summary reports `secret values printed: no`
- archive excludes iOS, Web, server, homepage, private detector, generated
  build, `.gradle`, `.DS_Store`, and `local.properties` content
- `leona-sdk-android/private/README.md` placeholder is the only private-path
  Android file included

## 3. Public Archive Consumer Smoke

Run:

```bash
./scripts/verify-v0.4-public-archive-consumer.sh
```

Required result:

- status is `pass`
- archive is extracted under `/tmp` or another explicit output directory
- required public Android files are present in the extracted archive
- extracted archive has no symlinks
- archived shell scripts pass `bash -n`
- extracted archive still excludes iOS, Web, server, homepage, generated local
  files, and private Android files other than `private/README.md`
- extracted archive scan does not find forbidden public-boundary material

## 4. Publish Workflow Dry Run

Run:

```bash
./scripts/verify-v0.4-publish-workflow-dry-run.sh
```

Required result:

- status is `local-pass-with-external-blockers`
- expected AAR and `.sha256` asset names match the SDK version
- workflow still runs on `v*` tags
- workflow has `contents: write` and `packages: write`
- tag jobs publish GitHub Release assets and GitHub Packages artifacts
- workflow does not reference Central, provider, customer, or deployment
  secrets
- remaining blockers are explicitly tag-time or external publish inputs

This dry-run does not create a tag, trigger GitHub Actions, or publish
artifacts.

```bash
  ./scripts/verify-v0.4-release-candidate-manifest.sh
```

Required result before pushing `v0.4.0`:

- status is `local-pass-with-external-blockers`
- require pre-tag ready is `1`
- `Pre-Tag Blockers` is `none`
- failures are `none`
The focused version marker gate must also pass with `VERSION_NAME=0.4.0`.

## 6. Tag Release Runbook

Before pushing a real tag, review
[`v0.4-tag-release-runbook.md`](v0.4-tag-release-runbook.md). The runbook fixes
the required order:

- bump `VERSION_NAME` to `0.4.0`
- review the read-only version bump plan
- run the read-only version bump dry-run
- run the focused version marker gate in strict mode
- rerun the release candidate manifest until `Pre-Tag Blockers` is `none`
- push annotated tag `v0.4.0`
- verify the tag workflow
- rerun post-release consumption smoke
- call out any external blockers that are intentionally deferred

## 9. Optional Heavy Gates

Run these only when the local machine or CI runner has the required toolchain
and network access:

```bash
LEONA_RUN_PUBLIC_GRADLE_GATE=1 ./scripts/verify-v0.4-release-readiness.sh
LEONA_RUN_PUBLIC_CONSUMPTION=1 ./scripts/verify-v0.4-release-readiness.sh
```

The Gradle gate validates the public SDK and sample build path. The consumption
gate validates published artifacts after a release exists.

## 10. Release Artifact Paths

Acceptable public artifact paths:

- GitHub Release AAR plus `.sha256`
- GitHub Packages Maven coordinate
- Maven Central coordinate after namespace, token, and signing material are
  ready
- public backend wrapper skeleton packages for customer server-side use

Maven Central is not complete until Central Portal namespace approval,
publishing credentials, and signing material are configured in a non-public
environment.

## 11. External Blockers

Do not mark v0.4 production-ready until these are either completed or explicitly
called out as deferred:

- custom ROM, GSI, unlocked bootloader, external emulator, and additional
  hide-module samples
- real Play Integrity or OEM attestation provider smoke
- Maven Central namespace, publishing token, and signing material
- authenticated live operations smoke with deployment-provided credentials
- real backend wrapper endpoint smoke using server-side SecretKey material

Synthetic credentials, debug provider responses, or dry-run evidence do not
close these blockers.

## 12. Privacy and Public Boundary Review

Before tagging, verify public artifacts do not include:

- full BoxIds
- raw Android ID, install ID, ADB serial, hardware serial, IMEI, IMSI, MAC
  address, advertising ID, or build fingerprint
- raw AppKey, tenant SecretKey, provider token, PGP key, signing passphrase, or
  deployment secret
- raw IP address or full package lists when a hash, hint, count, or family is
  sufficient
- private detector implementation details or customer-specific policy

Allowed forms include hashes, hints, counts, status values, family names, and
coarse evidence summaries.

## 12. Customer Backend Boundary

Customer backend wrappers are server-side helpers. They may sign requests,
fetch evidence reports, fetch support bundles, submit feedback labels, and
redact outputs.

They must not:

- run inside Android apps
- embed tenant SecretKey in APKs
- return business allow/reject/block decisions
- log full BoxIds, raw device identifiers, tokens, or provider credentials

## 13. Post-Release Consumption Smoke

After a tag and release are published, run:

```bash
LEONA_TARGET_RELEASE_VERSION=<released-version> \
  ./scripts/verify-v0.4-post-release-consumption.sh
```

Before the release exists, the wrapper writes a blocked summary rather than
pretending that a post-release artifact was consumed. After publication, rerun
with `LEONA_REQUIRE_POST_RELEASE_CONSUMPTION=1` to make GitHub Release AAR
download and checksum verification a hard gate. If a package-read token is
available in a non-public environment, the underlying consumer smoke also checks
the GitHub Packages Maven path. Maven Central consumption should be verified
only after Central publish is complete.
