# Leona Android SDK v0.4 Release Notes Draft

This draft is a public-safe release note template for the Android/Server v0.4
commercial pilot track. It must be reviewed again before tagging a real release.

## Positioning

v0.4 keeps the Android SDK evidence-only contract. The SDK collects and reports
device/environment evidence, returns an opaque `BoxId`, and leaves final
business decisions to the customer backend.

Leona evidence reports, Device Evidence Graph outputs, support bundles, feedback
reports, and backend wrappers are designed to help customer systems understand
provenance. They do not replace customer policy.

## Android SDK and Public Artifacts

Public-safe Android artifacts for the v0.4 track include:

- Android SDK public API and sample app
- GitHub Release AAR fallback
- GitHub Packages Maven path
- Maven Central readiness gate, pending external Central/PGP material
- backend wrapper skeletons for customer server-side use
- Android evidence/privacy boundary documentation
- v0.4 aggregate release readiness script
- public Android archive dry-run and archive consumer smoke
- Android tag publish workflow dry-run
- release candidate manifest for local pre-tag review
- release evidence pack that indexes local release-candidate component summaries
  and git index snapshots with SHA-256 digests, byte counts, and a
  public-boundary redaction scan
- release evidence pack schema gate that validates the generated JSON, expected
  component list, referenced file digests/byte counts, git index snapshots, and
  Markdown/JSON parity
- tag release runbook for version alignment, tag workflow, and post-release
  consumption smoke
- v0.4 post-release consumption wrapper that records pre-publication blocked
  state and becomes a strict artifact-consumption gate after tag publication
- Android version marker gate for focused `VERSION_NAME` / runtime marker
  review before the final `v0.4.0` tag
- Android version bump plan gate that previews required runtime marker updates,
  optional sample/docs preview updates, replacement counts, and planned
  SHA-256 digests without editing files
- Android version bump dry-run gate that applies those marker replacements in
  memory and proves required markers converge to the target version
- public commit scope gate for preventing mixed iOS/Web/server/homepage/private
  work from being staged into Android public GitHub commits

## Evidence and Graph Capabilities

The v0.4 server-side track adds or prepares:

- Device Evidence Graph timeline and velocity views
- evidence family aggregation for emulator, root, hook, hide, attestation,
  tamper, transport diagnostic, ROM/build posture, and install provenance
- customer evidence report and support bundle contracts
- feedback label schema and offline feedback evaluation reports

The Android SDK does not expose business `allow`, `reject`, `block`, or
`isFraud` decisions.

## Readiness Gates

Run the aggregate public-safe gate before release candidate review:

```bash
./scripts/verify-v0.4-release-readiness.sh
```

The gate validates local public-safe checks and reports external blockers. It
does not start paid devices, access private deployment systems, or print
credentials.

The archive gates validate both sides of the public release artifact:

```bash
./scripts/verify-v0.4-public-archive.sh
./scripts/verify-v0.4-public-archive-consumer.sh
```

The first command builds the temporary public Android archive and manifest. The
second command extracts that archive as a consumer would receive it, verifies
required public files, rejects private roots and generated local files, checks
archived shell-script syntax, and scans the extracted artifact for forbidden
public-boundary material.

The publish workflow dry-run checks tag-time GitHub Release and GitHub Packages
publication wiring without creating a tag or publishing artifacts:

```bash
./scripts/verify-v0.4-publish-workflow-dry-run.sh
```

The release candidate manifest composes the local readiness, archive consumer,
publish workflow dry-run, public commit scope, and public release batch
summaries into one pre-tag review artifact:

```bash
./scripts/verify-v0.4-release-candidate-manifest.sh
```

For the final tag check, run strict pre-tag mode:

```bash
LEONA_REQUIRE_PRETAG_READY=1 LEONA_TARGET_RELEASE_VERSION=0.4.0 \
  ./scripts/verify-v0.4-release-candidate-manifest.sh
```

The generated summary can be checked with
`verify-v0.4-release-candidate-manifest-schema.py` to validate required fields,
component paths, no-tag/no-publish/no-stage guarantees, public batch fields,
index preservation, and version-blocker semantics.

`verify-v0.4-release-candidate-review.sh` runs both the manifest and the schema
gate, then writes a combined summary that preserves the underlying failure
semantics.

`verify-v0.4-release-candidate-review-schema.py` validates that combined review
summary, including manifest/schema summary paths, wrapper failure semantics,
and no-tag/no-publish/no-stage/no-device guarantees.

`verify-v0.4-release-candidate-final-review.sh` runs the review wrapper and the
review schema gate together as the final local pre-tag review command.

`verify-v0.4-release-candidate-final-review-schema.py` validates that final
summary, including nested review/schema summary paths, final wrapper failure
semantics, and no-tag/no-publish/no-stage/no-device guarantees.

Strict mode fails while pre-tag blockers remain, such as a `VERSION_NAME` that
has not yet been bumped to `0.4.0`.

The manifest records public release batch counts, do-not-stage counts, staged
forbidden path counts, stage-command draft syntax, stage-command draft path
count, SHA-256 digests for the reviewed path lists, whether the draft exactly
matches the public batch list, and whether the draft contains any do-not-stage
path. It records `git-index-before.txt` and `git-index-after.txt`, then fails if
the staged index changes during the full release candidate run. It does not
execute real `git add` or stage files. Use the generated batch planner report
only as a review aid before intentional staging.

The generated public release batch report is also checked by
`verify-v0.4-public-release-batch-schema.py`. That gate independently verifies
path-list parity, SHA-256 digests, do-not-stage exclusion, forbidden public
prefixes, and dry-run git index preservation before the batch is included in
the release candidate evidence pack.

The manifest also writes `release-evidence-pack.md` and
`release-evidence-pack.json`. The pack indexes the readiness, archive consumer,
publish workflow dry-run, public commit scope, public release batch component
summaries, and before/after git index snapshots with paths, SHA-256 digests, and
byte counts. The redaction scan covers component summaries, git index snapshots,
and the generated evidence pack Markdown/JSON itself before the final pack is
rewritten with the final scan status. It is a local pre-tag review artifact, not
a release asset, tag, commit, device run, or publication step.

The generated JSON is then checked by
`verify-v0.4-release-evidence-pack-schema.py`. That gate verifies required
fields, the exact component set, referenced file SHA-256/byte counts, git index
snapshots, and Markdown/JSON parity. It is deliberately structural: a synthetic
`redactionScan=failed` pack remains schema-valid so the negative test produces
only the redaction failure.

The manifest also includes the read-only output of
`verify-v0.4-version-bump-plan.py`. The plan lists required runtime markers,
optional sample/docs preview markers, replacement counts, current digests, and
planned digests for the target release version. It does not apply the bump; the
actual edit belongs in the final release commit.

`verify-v0.4-version-bump-dry-run.py` applies the same public Android marker
replacements in memory and verifies that required `VERSION_NAME`, runtime, and
sample markers converge to the target release version. It does not write files,
stage paths, create tags, publish artifacts, start devices, or print secrets.

For a synthetic negative test of the generated-pack redaction path, run the
manifest with `LEONA_RC_SELF_SCAN_TEST=1`. This injects a synthetic marker only
into the temporary generated Markdown pack and should fail the manifest with
`release evidence pack redaction scan: failed`.

Before pushing a real tag, follow:

```text
docs/v0.4-tag-release-runbook.md
```

After the tag workflow publishes artifacts, run:

```bash
LEONA_TARGET_RELEASE_VERSION=0.4.0 \
  LEONA_REQUIRE_POST_RELEASE_CONSUMPTION=1 \
  ./scripts/verify-v0.4-post-release-consumption.sh
```

Before the release exists, the same wrapper records a blocked summary rather
than pretending that real artifacts were consumed.

Before staging or committing from a mixed local workspace, run:

```bash
./scripts/verify-v0.4-public-commit-scope.sh
```

The gate fails on staged non-public paths and reports unstaged or untracked
non-public work as material that must stay out of public Android commits.

The public release batch planner turns that scope output into reviewable path
lists and a syntax-checked, non-executed stage-command draft:

```bash
./scripts/verify-v0.4-public-release-batch.sh
```

The planner also extracts the draft paths back from the generated script,
records SHA-256 digests for the path lists, and fails if the draft differs from
the public batch list or intersects the do-not-stage list. It also runs the
generated draft in `LEONA_ANDROID_STAGE_DRY_RUN=1` mode and verifies the git
index is unchanged after `git add --dry-run`. It does not perform real staging,
commit, tag, publish, start devices, or print secrets.

Optional heavier checks:

```bash
LEONA_RUN_PUBLIC_GRADLE_GATE=1 ./scripts/verify-v0.4-release-readiness.sh
LEONA_RUN_PUBLIC_CONSUMPTION=1 ./scripts/verify-v0.4-release-readiness.sh
```

## External Blockers

The following items are not considered complete until real external material is
provided and verified:

- custom ROM / GSI / unlocked bootloader samples
- Nox, LDPlayer, BlueStacks, Genymotion, or equivalent external emulator samples
- additional hide-module combinations beyond the currently recorded matrix
- real Play Integrity or OEM attestation provider smoke
- Sonatype Central Portal namespace/token and PGP signing material
- authenticated live ops smoke with formal connector/report environment
- pilot backend wrapper endpoint smoke with server-side SecretKey

## Privacy and Public-Safe Rules

Release notes, ledgers, support bundles, scripts, and readiness summaries must
not include:

- full `BoxId`
- raw Android ID, install ID, ADB serial, hardware serial, IMEI, IMSI, MAC
  address, advertising ID, or build fingerprint
- raw AppKey, tenant SecretKey, provider token, PGP key, signing passphrase, or
  deployment secret
- raw IP address or full package list when a hash, hint, count, or family is
  sufficient

## Explicit Non-Goals

v0.4 Android release notes must not claim:

- Leona makes customer business decisions
- the Android app should trust local detector booleans for enforcement
- Maven Central publish is complete before Central/PGP material exists
- real attestation provider smoke is complete before provider material exists
- external emulator/custom ROM/GSI matrix is complete before real samples exist
- private detector, server, homepage, policy, or deployment internals are open
  source
