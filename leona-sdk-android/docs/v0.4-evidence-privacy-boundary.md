# Leona v0.4 Android Evidence and Privacy Boundary

This document defines the public-safe Android/Server boundary for the v0.4
commercial pilot track. It is intended for Android SDK integrators and customer
backend engineers.

## Evidence-Only Contract

The Android SDK only collects and reports device/environment evidence. It does
not make final business decisions and it must not expose APIs that tell an app
to allow, reject, block, challenge, or throttle a user.

Allowed Android SDK outputs:

- opaque `BoxId`
- diagnostic status for integration and transport debugging
- redacted support bundle fields
- evidence upload success/failure metadata

Not allowed in the Android SDK:

- business decision methods such as `allow`, `reject`, `block`, or `isFraud`
- tenant `SecretKey` or server-side signing material
- Play Integrity, OEM provider, Maven Central, or deployment credentials
- full raw device identifiers in logs, support bundles, or public docs

Customers should send the opaque `BoxId` to their own backend. The customer
backend can query Leona hosted evidence APIs, inspect evidence and provenance,
then apply the customer's own product policy.

## Evidence Families

v0.4 Android evidence is grouped for server-side reporting and graph analysis.
These families are facts or diagnostics, not client-side verdicts.

- emulator and virtualized runtime evidence
- Root, Magisk, Zygisk, LSPosed, HMA, and hide-module provenance
- hook, Frida, native mapping, and runtime injection provenance
- APK tamper, signature, installer, and packaging provenance
- ROM, GSI, verified boot, bootloader, build tag, and debug posture facts
- attestation provider status, freshness, and failure reason
- transport diagnostics such as TLS, timestamp, authentication, retry, and
  server response categories

The service may aggregate these families into Device Evidence Graph timelines,
velocity windows, customer evidence reports, and feedback evaluation reports.
Those reports must preserve source, trust, and provenance metadata.

## Redaction Requirements

Public docs, scripts, support bundles, ledgers, and release readiness summaries
must not expose:

- full `BoxId`
- raw Android ID, install ID, ADB serial, hardware serial, IMEI, IMSI, MAC
  address, advertising ID, or build fingerprint
- raw AppKey, tenant SecretKey, token, provider credential, PGP key, or signing
  passphrase
- raw IP address or deployment host secret
- complete package lists when a hash, count, family, or hint is enough

Use hash, hint, count, bucket, family, or status fields instead. A short
`BoxId` hint such as `01KS...ABCD` is acceptable for operator correlation, but
full identifiers are not acceptable in public artifacts.

## Customer Backend Wrapper Boundary

Backend wrappers under `leona-sdk-android/wrappers/` are server-side helpers.
They can sign requests, fetch evidence reports, fetch support bundles, submit
feedback labels, and redact outputs.

They must not:

- run inside Android apps
- embed tenant `SecretKey` in APKs
- return business `allow`, `reject`, or `block` decisions
- log full `BoxId`, raw device IDs, tokens, or provider credentials

The customer backend remains responsible for business policy and user-facing
actions.

## Release Readiness Boundary

`scripts/verify-v0.4-release-readiness.sh` is a public-safe aggregate gate for
the Android/Server v0.4 track. By default it can validate local public-safe
checks and report external blockers, but it does not start paid devices or print
secrets.

Default local checks include:

- public Android CI structure
- README evidence-only wording
- clean OEM ledger safety
- v0.4 Android matrix readiness ledger coverage
- Maven Central readiness metadata
- backend wrapper mock HTTP smoke

External blockers remain explicit until the required material exists:

- full external emulator, custom ROM/GSI/unlocked, and hide-module samples
- real Play Integrity or OEM provider credentials and allowlist
- Sonatype Central Portal namespace/token and PGP signing material
- authenticated live ops credentials and formal connector environment
- pilot endpoint and backend-only SecretKey for real wrapper endpoint smoke

These blockers should be recorded as external prerequisites, not papered over
with dummy credentials or synthetic success.
